Discovery 19: Configure a Provider-Assigned IPv4 Address
Enterprise Internet Connectivity Options
Modern enterprise IP networks are connected to the global internet and use it for some of their data communication needs. Enterprises provide many services to customers and business partners via the internet. Whether accessing resources in a cloud, connecting regional offices, or sharing web services with your customers, you must consider the options available to you. You should also understand how your company plans to use its internet connection.
The internet connection is established via an ISP. At the enterprise site, the router is the device used to connect to the ISP.
The figure illustrates enterprise internet connectivity requirements. Based on the direction of internet traffic, the connectivity requirements can be classified as outbound and inbound. For successful communication, an enterprise must ensure adequate resources for both outbound and inbound traffic.
The ISP can provide internet connectivity by providing statically assigned IPv4 addresses, or dynamically allocate them with DHCP. Depending on the option that is used, you must configure the internet-facing interfaces accordingly.
Statically assigned IPv4 addresses are useful in cases where a company or organization wants to have its servers or services publicly accessible. These static addresses can be linked to a domain name, such as http://www.cisco.com, which allows clients to find and access these servers and services.
Configuring a static provider-assigned IPv4 address is a two-step process. The first step of the process is to assign a static IPv4 address on the internet-facing interface of the router. The second step is to configure a default route that will forward all traffic that is intended for the internet to the ISP. The ISP must provide you with a static IPv4 address that you can configure on the interface.
When dynamic assignment is used by the ISP, no manual assignment of addresses is needed. Instead, the router interface needs to be configured as a DHCP client. Other configuration parameters can also be obtained through DHCP, such as the default gateway address.
Here are the advantages of dynamic address assignment:
  • Reduced configuration time
  • Reduced probability of configuration errors
The figure illustrates how to configure the router interface to obtain an IPv4 address through DHCP.
Obtaining an IPv4 address through DCHP involves configuring a DHCP client, which will request IPv4 addressing information from the DHCP server:
  • The router requests IPv4 addressing information from the DHCP server.
  • The router injects a static default route in its routing table based on the default gateway parameters received with the assigned IPv4 address.

PC1 is not assigned an IPv4 address, although it is configured to receive information via DHCP. This situation is expected.
DHCP clients need to use User Datagram Protocol (UDP) broadcasts to send their initial DHCPDISCOVER messages, because they donâ€™t have information about the network to which they are attached to. If the client is on a network segment that does not include a server, UDP broadcasts are normally not forwarded, because most routers are configured to not forward broadcast traffic. So broadcast messages that are sent by PC1 arrive only to the Ethernet0/0 interface of R1 and do not cross the R1 router. Because the DHCP server is located on the R2 router, PC1 DHCP requests never reach the server.
You can remedy this situation by configuring the interface of your router that is receiving the broadcasts to forward certain classes of broadcast traffic to a specific IPv4 address, called helper address. You can use more than one helper address per interface. When configured with a helper address, the router is considered to be the relay agent.
To configure a helper address, use the ip helper-address IPv4-address interface configuration mode command.
When a helper address is configured for DHCP, the router "picks up" a broadcast packet from the interface it receives it on, and it generates new DHCP messages to send out to the indicated helper address. In the new DHCP message, the router (relay agent) inserts the IPv4 address of its interface, where broadcast message was received and where the ip helper-address command is configured, into the gateway IPv4 address (giaddr) field of the DHCP packet. This giaddr IPv4 address enables the DHCP server to determine which subnet should receive the offer and to identify the appropriate IPv4 address pool from which to select the address. The DHCP relay agent forwards the received broadcast as a unicast packet, to the address of the DHCP server specified by the ip helper-address command.
Note
A DHCP relay agent is any host that forwards DHCP packets between clients and servers. Relay agents are used to forward requests and replies between clients and servers when they are not on the same physical subnet. Relay agent forwarding is distinct from the normal forwarding, where IP packets are switched somewhat transparently. By contrast, relay agents receive DHCP messages and then generate a new DHCP message to send out on another interface.
Step 10: Configure the R1 router to relay DHCP broadcasts.
On R1, enter the following command:
R1# configure terminal R1(config)# interface Ethernet0/0 R1(config-if)# ip helper-address 198.51.100.1 R1(config-if)# end
The 198.51.100.1 address, specified as a helper address, is the address of the R2 router. R2 is the DHCP server for the 10.10.1.0/24 subnet.
All broadcasts that R1 receives on its Ethernet0/0 interface will be forwarded to the 198.51.100.1 address as unicast messages. The router will add the 10.10.1.1 IPv4 address information to each of the unicast messages. It will add the address into the giaddr field of the DHCP message.

The show dhcp commands verify the basic DHCP IPv4 address and default gateway information, along with DHCP-specific information. Step 13: R2 is configured as the DHCP server for both the 198.51.100.0/24 and 10.10.1.0/24 segments. Although the DHCP server configuration is beyond the scope of this course, you can view the configuration, to understand what information is required to be configured on the DHCP server. DHCP configuration lines are present in the running configuration and startup configuration file of R2. To filter out only DHCP-related information from the startup configuration file, use the show startup-config | section dhcp command.
On R2, enter the following command:
R2# show startup-config | section dhcp ip dhcp excluded-address 198.51.100.1 198.51.100.100 ip dhcp excluded-address 10.10.1.1 10.10.1.100 ip dhcp pool ClientRouter network 198.51.100.0 255.255.255.0 default-router 198.51.100.1 lease 7 ip dhcp pool Client_10_10_1_0 network 10.10.1.0 255.255.255.0 default-router 10.10.1.1 lease 5
IPv4 parameters meant to be assigned on the 198.51.100.0/24 segment are specified in the DHCP pool ClientRouter. The pool specifies that addresses from the range 198.51.10.0/24 should be offered to DHCP clients. The default gateway for the segment is 198.51.100.1, and the lease period is 7 days. The configuration also specifies, in the ip dhcp excluded-addresses command, that addresses from 198.51.100.1 to 198.51.100.100 should not be offered. Therefore, the R1 router has received the address 198.51.100.101, the first available address from the ClientRouter pool. Address 198.51.100.1 is excluded, because it is already statically assigned to the R2 router Ethernet 0/3 interface, and it should not be assigned to any other host.
IPv4 parameters meant to be assigned on the 10.10.1.0/24 segment are specified in the DHCP pool Client_10_10_1_0. The pool specifies that addresses from the range 10.10.1.0/24 should be offered to DHCP clients. The default gateway for the segment is 10.10.1.1, and the lease period is 5 days. The configuration also specifies, in the ip dhcp excluded-addresses command, that addresses from 10.10.1.0 to 10.10.1.100 should not be offered. Therefore, PC1 has received the 10.10.1.101 address, the first available address from the Client_10_10_1_0 pool. The 10.10.1.1 address is excluded, because it is already statically assigned to the R1 router Ethernet 0/0 interface and it should not be assigned to any other host in the 10.10.1.0/24 segment.
