A basic network security policy was configured on devices by your colleague, but you can still see that the configurations are incomplete as per the standard basic security policy that is adopted by the CCS company for the customer environment.
The security policy that is adopted by CCS for R1, the Branch router, and SW1 is as follows:
  • Access to the privileged mode must be secured using a password on all devices.
  • Passwords must be encrypted on all devices.
  • Access to the console and vty lines must be secured on all devices.
  • Remote access must be secured by enabling SSH on vty 0 to 4 on all devices.
  • A standard IPv4 ACL must be used to restrict remote access on all devices. The ACL should allow the traffic from only the AdminPC.
  • A login banner must be present on all devices.
Here are the issues that you need to troubleshoot and resolve:
  • Local authentication is not working on R1.
  • AdminPC is not able to SSH to R1 IPv4 address 172.16.160.1. Generate a pair of RSA keys on R1 that will allow the router to support SSHv2. Use the domain name example.com and use 768 bits in the modulus.
  • The enable password should be encrypted on SW1 and the Branch router.
  • AdminPC is not able to SSH to the Branch router (172.16.160.3).
  • AdminPC is not able to SSH to SW1 (172.16.130.10). Generate a pair of RSA keys on SW1 that will allow the switch to support SSHv2. Use the domain name example.com and use 768 bits in the modulus.
Refer to Job Aids for login credential information.
Note
The Branch router may take 3-5 minutes to boot up and become accessible.

If you shut down an interface on a real router or switch, the connected device will see it as "down/down." Due to virtualization specifics, Cisco IOL (Cisco IOS Software on Linux) behavior is slightly different. If you shut down an interface on a router or switch, the connected device will see it as "up/up." In Cisco IOL, the status of an interface can only be "up/up" or "administratively down/down."

Answer Key
Answer Key
The security policy that is adopted by CCS for R1, the Branch router, and SW1 is as follows:
  • Access to the privileged mode must be secured using a password on all devices.
    Login in to R1, Branch, and SW1. Enter privileged mode using the enable command; you should be asked for a password.
  • Passwords must be encrypted on all devices.
    On each device verify that passwords are encrypted
  • Access to the console and vty lines must be secured on all devices.
    Try logging in to R1, Branch, and SW1 console and vty lines; you should be asked for credentials.
  • Remote access must be secured by enabling SSH on vty 0 to 4 on all devices.
    Try logging in to R1, Branch, and SW1 vty lines using SSH.
  • A standard IPv4 ACL must be used to restrict remote access on all devices. The ACL should allow the traffic from only the AdminPC.
    Only the AdminPC should be able to login in to R1, Branch, and SW1 vty lines using SSH.
  • A login banner must be present on all devices.
    Try logging in to R1, Branch, and SW1 console and vty lines; you should see a login banner.
Here are the issues that you need to troubleshoot and resolve:
  • Local authentication is not working on R1.
    Execute the show running-configuration | begin line command on R1 to verify that the local authentication is configured.
    R1#show running-configuration | begin line Building configuration... Current configuration : 131 bytes ! ! ! ! ! ! ! line con 0 logging synchronous line aux 0 line vty 0 4 access-class 11 in no login transport input telnet ! end
    You will notice that the local authentication is not configured on R1.
    Verify that a username and password is configured on R1.
    R1#show running-configuration | include username username admin password 7 072E25414707485744
    Execute the following commands to enable the local authentication on the console and vty lines.
    R1# configure terminal Enter configuration commands, one per line. End with CNTL/Z. R1(config)# line con 0 R1(config-line)# login local R1(config-line)# line vty 0 4 R1(config-line)# login local R1(config-line)# end
    Log out and then login again on R1 to verify the local authentication is working.
    !!! WARNING: ACCESS TO THIS DEVICE IS ONLY FOR AUTHORIZED PERSONNEL!!! User Access Verification Username: admin Password: R1>
  • AdminPC is not able to SSH to R1 IPv4 address 172.16.160.1. Generate a pair of RSA keys on R1 that will allow the router to support SSHv2. Use the domain name example.com and use 768 bits in the modulus.
    Notice from the output in the previous step that R1 is not configured to support SSH. Generate a pair of RSA keys on R1 that will allow the router to support SSHv2. Use the domain name example.com and use 768 bits in the modulus. Configure the vty lines to support SSH and configure SSHv2. Execute the following commands on R1:

    Execute the show access-list command on Branch router to verify the access-list configured on Branch router.

    Branch# configure terminal Enter configuration commands, one per line. End with CNTL/Z. Branch(config)# no access-list 11 Branch(config)# access-list 11 permit 172.16.130.5
    Execute the ssh -l admin 172.16.160.3 command on Admin PC to verify that the AdminPC is now able to SSH to the Branch router. Terminate the SSH session and return to the Admin PC.

Questions
Questions
Question 1
Which of the following commands do you use to check if access to privileged mode is secured with a password? (Choose two.)
The correct answer is "show running-config | include enable." and "show running-config | section enable." The enable command in the global configuration mode restricts the access to the privilege mode with the password that is provided with the command.
Question 2
Which two need to be configured on R1 for local authentication? (Choose two)
The correct answers are: "login local on vty line" and "login local on console line". The R1 router does have the username and password configured, but it is missing the login local command in the console and vty lines section to authenticate users accessing the device from the console or over the SSH connection.
Question 3
Which is the first command you have to enter on the Branch router when correcting the issue that the AdminPC could not SSH to the Branch router?
The correct answer is: "Branch(config)# no access-list 11." The Branch router is configured with the access list with the number 11 which denies the traffic from the AdminPC. Adding an access list with a greater number of 11, which permits the access from an IP, does not get registered since access lists check the rules from lower numbers towards higher numbers. Access lists 1 and 10 do not exist, configuring access list without a number is not a valid command.
Question 4
Which option describes what is needed to configure SSHv2 on SW1?
The correct answer is: "Configure the domain name, generate the RSA keys, add SSH to vty lines, and globally enable SSHv2." To access SW1 via SSH, the mentioned configuration needs to be applied. The switch has already a permit access list defined for the AdminPC and a username with the corresponding password.
